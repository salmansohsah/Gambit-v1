## 0. Core Architecture Principle (Non-Negotiable)

This must be understood before reading any other section.

```
┌─────────────────────────────────────────────────────────────┐
│              SOURCE OF TRUTH ARCHITECTURE                   │
│                                                             │
│  GitHub    →  Source of Truth for CODE                      │
│  Supabase  →  Source of Truth for CONTENT                   │
│  Vercel    →  Deployment Layer (reads code from GitHub)     │
│  Admin CMS →  Content Management Layer (writes to Supabase) │
│                                                             │
│  RULE: Content changes NEVER require Git commits.           │
│  RULE: Code changes NEVER go through Admin CMS.             │
│  RULE: These two flows are completely separate.             │
└─────────────────────────────────────────────────────────────┘
```

### What this means practically:
- Admin edits a page headline → writes to Supabase → Next.js cache revalidated → live immediately. No GitHub involved.
- Developer adds a new feature → commits to GitHub → Vercel deploys. No Supabase writes involved.
- The Sync Center is a READ-ONLY status dashboard. It reads from GitHub/Vercel APIs. It does NOT write to GitHub.
- Content snapshots are stored in Supabase, not in Git.

## 7. Navigation Architecture

### Table: `navigation_items`

```sql
CREATE TABLE navigation_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  menu_type TEXT NOT NULL CHECK (menu_type IN ('header', 'footer', 'legal', 'social')),
  label TEXT NOT NULL,
  href TEXT NOT NULL,
  display_order INTEGER NOT NULL DEFAULT 0,
  is_external BOOLEAN DEFAULT FALSE,
  is_active BOOLEAN DEFAULT TRUE,
  icon_name TEXT,                    -- for social links (e.g., 'instagram', 'x', 'linkedin')
  target TEXT DEFAULT '_self',       -- '_blank' for external
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

**menu_type values:**
- `header` — top navigation bar
- `footer` — footer link columns
- `legal` — bottom footer (Privacy Policy, Terms)
- `social` — social media icons/links (replaces current placeholder)

**Why one table, not four:**
- Reduces join complexity
- Filter by `menu_type` for each use case
- Admin can manage all menus from one screen with tab navigation
- Future menu types (e.g., `mobile`, `sidebar`) require only a new enum value

**Frontend wiring:**
- `Navigation.tsx` — reads `header` items, with hardcoded fallback
- Footer component — reads `footer` + `legal` + `social` items
- Social icons in RecentMovesSection will read from `social` type

## 9. Audit Logging Architecture

Every Admin write operation must be traceable.

### Table: `audit_logs`

```sql
CREATE TABLE audit_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  user_email TEXT,
  action TEXT NOT NULL,               -- 'CREATE', 'UPDATE', 'DELETE', 'PUBLISH', 'ARCHIVE', 'RESTORE'
  entity_type TEXT NOT NULL,          -- 'project', 'insight', 'page_content', 'navigation', 'seo', etc.
  entity_id TEXT,                     -- UUID or string identifier
  entity_label TEXT,                  -- human-readable name of the entity
  previous_value JSONB,               -- snapshot of data BEFORE change
  new_value JSONB,                    -- snapshot of data AFTER change
  ip_address TEXT,
  user_agent TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Indexes for common queries
CREATE INDEX idx_audit_logs_user_id ON audit_logs(user_id);
CREATE INDEX idx_audit_logs_entity_type ON audit_logs(entity_type);
CREATE INDEX idx_audit_logs_created_at ON audit_logs(created_at DESC);
```

**Implementation approach:**
- Server Actions in `/app/actions/` call a shared `logAuditEvent()` utility AFTER each successful mutation
- `logAuditEvent()` is a fire-and-forget (non-blocking) Supabase insert
- The audit log is APPEND-ONLY — no UPDATE or DELETE on `audit_logs`
- Admin can view audit logs at `/admin/audit` (read-only table, filterable by entity type, user, date)
- RLS: Admin-only read access. No user can delete audit logs.

## 10. Content Lifecycle Management

### Lifecycle States

```
DRAFT → REVIEW → PUBLISHED → ARCHIVED
         ↓           ↓
      (reject)    (unpublish)
         ↓           ↓
        DRAFT      ARCHIVED
```

### Applies to:
- `insights` (already has `status` field — verify enum values)
- `projects` (already has `status` field — verify enum values)  
- `page_content` (new: add `draft_value` column for staging edits)

### Database Changes:

**For `insights` and `projects`:**
Verify existing `status` field accepts: `'draft'`, `'review'`, `'published'`, `'archived'`

**For `page_content` (new columns):**
```sql
ALTER TABLE page_content 
  ADD COLUMN draft_value_text TEXT,
  ADD COLUMN draft_value_json JSONB,
  ADD COLUMN reviewed_by UUID REFERENCES auth.users(id),
  ADD COLUMN reviewed_at TIMESTAMPTZ,
  ADD COLUMN published_at TIMESTAMPTZ,
  ADD COLUMN published_by UUID REFERENCES auth.users(id);
```

### Workflow Logic:
- **Draft:** Content is saved but not live. Frontend reads `value_text`/`value_json` only.
- **Review:** Content submitted for approval. Reviewer can approve (→ Published) or reject (→ Draft).
- **Published:** `draft_value` is merged into `value_text`/`value_json`. Frontend immediately reflects.
- **Archived:** Content is hidden from frontend but preserved in DB + audit log.

## 11. Revision History Architecture

Every content item should support viewing and restoring previous versions.

### Option A — Dedicated Revisions Table (Recommended)

```sql
CREATE TABLE content_revisions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_type TEXT NOT NULL,          -- 'project', 'insight', 'page_content'
  entity_id TEXT NOT NULL,
  revision_number INTEGER NOT NULL,
  snapshot_json JSONB NOT NULL,       -- full row snapshot at time of change
  change_summary TEXT,                -- auto-generated or admin-written
  created_by UUID REFERENCES auth.users(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_content_revisions_entity ON content_revisions(entity_type, entity_id, revision_number DESC);
```

**Behavior:**
- A revision is created BEFORE every UPDATE to a content item (pre-save snapshot)
- Admin can view revision history at `/admin/portfolio/{id}/history`
- Admin can compare two revisions (JSON diff view)
- Admin can restore a revision (writes old snapshot back to live table + creates new revision + creates audit log entry)
- Revisions are capped at 50 per entity (oldest pruned automatically)

### Option B — Leverage audit_logs (Simpler, but limited)
Use `previous_value` in `audit_logs` as a de-facto revision store. Cheaper but harder to query/diff.

**Decision: Option A recommended** — dedicated table allows proper UI and efficient queries.

## 12. Snapshot System Architecture (Expanded)

### Table: `content_snapshots`

```sql
CREATE TABLE content_snapshots (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  label TEXT NOT NULL,
  snapshot_type TEXT NOT NULL CHECK (snapshot_type IN ('manual', 'auto_pre_publish', 'auto_pre_restore', 'auto_scheduled')),
  snapshot_json JSONB NOT NULL,       -- captures: page_content, site_settings, navigation_items, seo_overrides
  entity_counts JSONB,                -- e.g., {"page_content": 42, "navigation_items": 8}
  created_by UUID REFERENCES auth.users(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  notes TEXT
);
```

### Snapshot Types:
| Type | Trigger |
|---|---|
| `manual` | Admin clicks "Create Snapshot" in Sync Center |
| `auto_pre_publish` | Automatically before any bulk publish action |
| `auto_pre_restore` | Automatically before restoring any prior snapshot |
| `auto_scheduled` | Future: cron-triggered (daily/weekly) — documented as future scope |

### Snapshot Scope:
A snapshot captures the entire content state:
- All `page_content` rows
- `site_settings` row
- All `navigation_items` rows
- All `seo_overrides` rows

A snapshot does NOT capture:
- `projects` (managed as individual revisions)
- `insights` (managed as individual revisions)
- `audit_logs` (never modified, always append-only)

### Restore Process:
1. Create `auto_pre_restore` snapshot of current state
2. Write snapshot_json data back to respective tables
3. Call `revalidateTag` for all affected cache tags
4. Log restore action to `audit_logs`
5. Display confirmation in admin

### Export:
- Download snapshot as `.json` file (browser download)
- Useful for migrating content or debugging

## 14. Content Type Registry Architecture

A centralized registry to define what content types exist in the system. This allows future content types to be added without restructuring the CMS.

### Table: `content_type_registry`

```sql
CREATE TABLE content_type_registry (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  type_key TEXT NOT NULL UNIQUE,       -- e.g., 'hero', 'services', 'case_study', 'insight'
  label TEXT NOT NULL,                 -- Human-readable: 'Hero Section'
  description TEXT,
  admin_route TEXT,                    -- e.g., '/admin/page-content'
  supports_drafts BOOLEAN DEFAULT TRUE,
  supports_revisions BOOLEAN DEFAULT TRUE,
  supports_seo BOOLEAN DEFAULT FALSE,
  is_singleton BOOLEAN DEFAULT FALSE,  -- true for Hero (only one), false for Case Studies
  schema_json JSONB,                   -- optional: field definitions for dynamic forms
  display_order INTEGER DEFAULT 0,
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### Supported Types (initial seed):
| type_key | label | singleton | SEO |
|---|---|---|---|
| `hero` | Hero Section | true | false |
| `services` | Services / What We Do | true | false |
| `case_study` | Case Study | false | true |
| `insight` | Insight / Article | false | true |
| `testimonial` | Testimonial | false | false |
| `navigation` | Navigation Item | false | false |
| `page_content` | Page Content Block | false | false |
| `seo_override` | SEO Override | false | true |
| `team_member` | Team Member | false | false |

**Why this matters:**
- Admin sidebar can be dynamically generated from this table
- New content types can be added by inserting a row — no code changes
- Future: dynamic form builder can read `schema_json` to render edit forms

## 17. Rollback Strategy

1. **Homepage IA change:** Revert `page.tsx` — 1 file, 1 git revert. WhatWeDoSection file can remain.
2. **New admin pages:** Delete new route directories — no impact on existing pages.
3. **New DB tables:** DROP TABLE in reverse order (respecting FK constraints). No existing data touched.
4. **Content changes:** Use snapshot restore to revert any content to last known good state.
5. **Git:** Atomic commits per Sprint — `git revert <commit>` per sprint.
