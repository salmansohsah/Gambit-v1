## 0. Core Architecture Principle (Non-Negotiable)

```
┌─────────────────────────────────────────────────────────────┐
│              SOURCE OF TRUTH ARCHITECTURE                   │
│                                                             │
│  GitHub    →  Source of Truth for CODE only                 │
│  Supabase  →  Source of Truth for CONTENT only              │
│  Vercel    →  Deployment Layer (deploys code from GitHub)   │
│  Admin CMS →  Content Management Layer (writes to Supabase) │
│                                                             │
│  RULE: Content changes NEVER require Git commits.           │
│  RULE: Code changes NEVER go through Admin CMS.             │
│  RULE: Sync Center is READ-ONLY — it NEVER writes to Git.   │
│  RULE: Content snapshots live in Supabase, not in Git.      │
└─────────────────────────────────────────────────────────────┘
```

## 1. Locked Decisions (Q1–Q6)

### Q1 — WhatWeDoSection Data Source ✅ LOCKED
- Reuses the existing `capabilities` table. No new data source created.
- `WhatWeDoSection` and `FrameworkSection` consume the same dataset with different visual presentations.
- All service management happens from `/admin/capabilities` only.
- If homepage-specific fields are ever needed, extend `capabilities` schema (e.g., add `homepage_description` column) — never create a parallel table.

### Q2 — Navigation Fallback Behavior ✅ LOCKED
- If `navigation_items` table is empty, returns an error, or hasn't been seeded:
  → Fall back silently to current hardcoded navigation array.
- Navigation must NEVER disappear due to missing CMS data.
- Log to console in dev mode: `[Navigation] Falling back to hardcoded nav — DB returned empty`
- Once DB has valid records, CMS data becomes the active source automatically.

### Q3 — Sync Center API Tokens ✅ LOCKED
- Sync Center renders in all cases.
- If `VERCEL_TOKEN` missing → show card: "Vercel API not configured. Add VERCEL_TOKEN to enable."
- If `GITHUB_TOKEN` missing → show card: "GitHub API not configured. Add GITHUB_TOKEN to enable."
- Architecture must allow enabling tokens later with zero code changes (env var only).

### Q4 — Testimonials Frontend ✅ LOCKED
- Testimonials Manager backend is built in Sprint 3.
- NO public homepage testimonial section during Sprint 1 or Sprint 2.
- Frontend rendering of testimonials is a Sprint 3 concern.

### Q5 — DB Migrations Strategy ✅ LOCKED WITH SAFEGUARDS
Migration Protocol (mandatory for every schema change):
1. Write migration SQL file → save to `/dev-backup/migrations/NNN_name.sql`
2. Write rollback SQL file → save to `/dev-backup/migrations/NNN_name_rollback.sql`
3. Present SQL to user for review
4. Create a content snapshot BEFORE applying
5. Apply migration ONLY after explicit approval
6. Confirm success before proceeding

Rules:
- Never apply a migration immediately after writing it
- Never drop existing tables without explicit approval
- Prefer additive schema changes (ADD COLUMN, CREATE TABLE)
- Treat production Supabase as high risk — incremental only
- Number migrations sequentially: `001_`, `002_`, etc.

### Q6 — Homepage Content Strategy ✅ LOCKED
- Target ratio: **80% business clarity, 20% strategic brand storytelling**
- `WhatWeDoSection` placed immediately after Hero — first thing visitors read after the headline
- A first-time visitor must understand GAMBIT's core offerings within the first screen and a half
- Do NOT use philosophy-heavy placeholder copy as the long-term strategy
- All fallback copy in components must reflect the 80/20 rule

### Schema Reference

#### `navigation_items`
```sql
id UUID PK, menu_type TEXT CHECK IN ('header','footer','legal','social'),
label TEXT, href TEXT, display_order INT, is_external BOOL DEFAULT false,
is_active BOOL DEFAULT true, icon_name TEXT, target TEXT DEFAULT '_self',
created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
```

#### `seo_overrides`
```sql
id UUID PK, page_identifier TEXT UNIQUE,
meta_title TEXT, meta_description TEXT, canonical_url TEXT,
robots TEXT DEFAULT 'index, follow',
og_title TEXT, og_description TEXT, og_image_url TEXT,
twitter_card TEXT DEFAULT 'summary_large_image',
twitter_title TEXT, twitter_description TEXT, twitter_image_url TEXT,
structured_data_json JSONB,
created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
```

#### `audit_logs`
```sql
id UUID PK, user_id UUID REF auth.users, user_email TEXT,
action TEXT ('CREATE','UPDATE','DELETE','PUBLISH','ARCHIVE','RESTORE'),
entity_type TEXT, entity_id TEXT, entity_label TEXT,
previous_value JSONB, new_value JSONB,
ip_address TEXT, user_agent TEXT, created_at TIMESTAMPTZ
-- Indexes: user_id, entity_type, created_at DESC
-- RLS: admin read-only, no delete
```

#### `content_revisions`
```sql
id UUID PK, entity_type TEXT, entity_id TEXT,
revision_number INT, snapshot_json JSONB, change_summary TEXT,
created_by UUID REF auth.users, created_at TIMESTAMPTZ
-- Index: (entity_type, entity_id, revision_number DESC)
-- Cap: 50 revisions per entity (oldest pruned)
```

#### `content_snapshots`
```sql
id UUID PK, label TEXT,
snapshot_type TEXT CHECK IN ('manual','auto_pre_publish','auto_pre_restore','auto_scheduled'),
snapshot_json JSONB, entity_counts JSONB,
created_by UUID REF auth.users, created_at TIMESTAMPTZ, notes TEXT
```

#### `content_type_registry`
```sql
id UUID PK, type_key TEXT UNIQUE, label TEXT, description TEXT,
admin_route TEXT, supports_drafts BOOL, supports_revisions BOOL,
supports_seo BOOL, is_singleton BOOL, schema_json JSONB,
display_order INT, is_active BOOL, created_at TIMESTAMPTZ
```

#### `testimonials`
```sql
id UUID PK, client_name TEXT, role TEXT, company TEXT,
quote TEXT, avatar_url TEXT, display_order INT,
is_active BOOL DEFAULT true, created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
```

## 6. Navigation Architecture (Locked)

Single `navigation_items` table with `menu_type` controlling context:
- `header` → Top navigation bar (`Navigation.tsx`)
- `footer` → Footer links
- `legal` → Bottom footer (Privacy, Terms)
- `social` → Social media icons/links

Frontend fallback chain:
```
DB query → if empty/error → hardcoded array → log warning in console
```

## 8. Audit Log Architecture (Locked)

- Shared utility: `lib/utils/auditLogger.ts` → `logAuditEvent()`

- Append-only table — no UPDATE or DELETE on audit_logs ever
- Admin view at `/admin/audit` — read-only, filterable

## 10. Snapshot Strategy (Locked)

Types: `manual`, `auto_pre_publish`, `auto_pre_restore`, `auto_scheduled`

Scope: `page_content` + `site_settings` + `navigation_items` + `seo_overrides`

Restore protocol:
1. Auto-create `auto_pre_restore` snapshot first
2. Write snapshot data to tables
3. Revalidate all cache tags
4. Write to audit_logs

## 11. Risk Assessment

| Risk | Severity | Mitigation |
|---|---|---|
| WhatWeDoSection layout clash | LOW | New component, no existing code touched |
| Navigation DB empty at launch | MEDIUM | Hardcoded fallback (Q2 locked) |
| Migration partially applied | HIGH | Atomic SQL, rollback file prepared |
| Snapshot restore overwrites live content | HIGH | Auto pre-restore snapshot always created first |
| Audit log inserts slow mutations | LOW | Fire-and-forget async, non-blocking |
| Revision table unbounded growth | LOW | 50-cap per entity, automatic pruning |

---

## 12. Rollback Strategy

1. **Homepage changes:** Revert `page.tsx` — 1 git revert
2. **WhatWeDoSection:** Component file can remain even if removed from page.tsx
3. **Admin pages:** Delete route directories — isolated, no side effects
4. **DB tables:** Execute rollback SQL files from `/dev-backup/migrations/`
5. **Content changes:** Restore from content snapshot in Sync Center
6. **Git strategy:** Atomic commit per sprint phase
